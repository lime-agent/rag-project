# RAG란 무엇인가

RAG는 Retrieval-Augmented Generation의 약자다. 직역하면 "검색으로 보강된 생성"이다.

## 왜 RAG가 필요한가

Claude 같은 LLM은 학습 데이터로 답을 만든다. 내가 오늘 만든 결정 노트, 오늘 겪은 오류 기록은 Claude가 모른다. RAG는 이 문제를 해결한다.

질문이 들어오면 먼저 내 문서에서 관련 내용을 검색하고, 그 내용을 Claude에게 함께 던진다. Claude는 내 문서를 참고해서 답한다.

## RAG 처리 흐름

```
질문 입력
  |
  v
질문 임베딩 (숫자 벡터로 변환)
  |
  v
ChromaDB에서 유사한 청크 검색 (top-3)
  |
  v
LLM에게: "아래 문서를 참고해서 답해줘" + 검색된 청크
  |
  v
LLM 답변 + 출처 파일명
```

## 임베딩이란

텍스트를 숫자 벡터로 변환하는 것이다. 의미가 비슷한 문장은 벡터 공간에서 가까운 위치에 놓인다. 이 거리를 계산해서 유사한 문서를 찾는다.

이 프로젝트에서는 Ollama의 `nomic-embed-text` 모델을 사용한다. 768차원 벡터를 생성하며 한국어를 포함한 다국어를 지원한다. Ollama가 이미 설치되어 있으면 `ollama pull nomic-embed-text` 한 줄로 추가 설치 없이 사용할 수 있다.

## ChromaDB란

벡터를 저장하는 데이터베이스다. 로컬 파일(`.chroma/`)에 저장되어 서버가 필요 없다. 질문 벡터와 가장 가까운 문서 청크를 빠르게 찾아준다.
